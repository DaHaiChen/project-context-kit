# 文档生成流程

本流程用于约定以后新增项目、业务域、大模块或子模块时，PRD、设计、QA、验收文档应该如何生成，以及哪些内容需要同步沉淀到 `.project-context/`。

## 1. 文档分层与轻量原则

项目文档分两层维护：

1. `docs/`：完整交付文档层。
2. `.project-context/`：项目语义索引与契约层。

`docs/` 负责保存完整业务过程和交付材料；`.project-context/` 负责给 agent 和跨模块协作者快速定位模块边界、契约、风险和影响面。

轻量原则：日常先写 `docs/`，不要每次草稿变化都同步 `.project-context/`。只有当 PRD/设计确认、契约变化、模块边界变化、架构决策成立或一轮开发/联调结束时，才执行 `/context-sync`。

## 2. 新模块标准流程

### Stage 1：需求共识

输出文件：

```text
docs/<业务域>/<模块>/需求共识.md
docs/<业务域>/<模块>/QA待确认.md
```

内容要求：

- 背景与目标。
- 包含范围与不包含范围。
- 角色、端侧、权限边界。
- 主流程、异常流程、关键规则。
- 待确认问题。

### Stage 2：大模块 PRD

输出文件：

```text
docs/<业务域>/<模块>/<模块>PRD.md
```

内容要求：

- 模块目标。
- 范围定义。
- 角色与权限。
- 统一业务流程。
- 统一状态机。
- 数据模型总览。
- 页面与交互总览。
- 接口策略。
- 子模块列表。
- 验收标准。
- 风险与待确认事项。

大模块 PRD 只写共性规则；子模块差异不在大模块里重复展开。

### Stage 3：子模块 PRD

输出文件：

```text
docs/<业务域>/<模块>/<子模块>.md
```

内容要求：

- 子模块目标。
- 与大模块对齐项。
- 差异化业务规则。
- 差异化字段。
- 差异化状态流转。
- 页面与交互。
- 端侧权限矩阵。
- 接口差异。
- 验收标准。
- 子模块开发审查清单。

原则：大模块管共性，子模块管差异。

### Stage 4：技术设计 / 一期设计

输出文件：

```text
docs/<业务域>/<模块>/<模块>一期设计.md
docs/<业务域>/<模块>/接口设计.md
```

内容要求：

- 设计目标。
- 实现范围。
- 架构关系。
- 前端设计。
- 后端设计。
- 接口设计。
- 数据库设计。
- 权限设计。
- 异常与边界。
- 联调说明。
- 测试建议。
- 风险。

数据库字段表统一使用以下结构：

| 字段名 | 中文名 | 字段类型 | 是否必填 | 说明 |
|---|---|---|---|---|
| `field_name` | 字段中文名 | `varchar(100)` | 是 / 否 | 业务含义、规则、枚举值、默认值 |

### Stage 5：上下文沉淀

PRD 或设计评审通过后，再从 `docs/` 提炼长期有效内容到 `.project-context/`。草稿阶段只维护 `docs/`，不做强制同步。

| docs 中的信息 | 沉淀位置 |
|---|---|
| 模块职责、不负责范围、关键流程、已知坑点 | `.project-context/modules/<子项目>/<模块>.md` |
| HTTP/RPC/内部服务接口语义 | `.project-context/contracts/api.md` |
| 表、字段、状态、枚举、删除校验、数据一致性 | `.project-context/contracts/database.md` |
| 权限码、菜单权限、按钮权限、角色边界 | `.project-context/contracts/permissions.md` |
| MQ、领域事件、异步通知、定时任务触发 | `.project-context/contracts/events.md` |
| 长期架构取舍和原因 | `.project-context/decisions/*.md` |
| 当前阶段重点、阻塞和短期风险 | `.project-context/active-work/*.md` |
| 模块、代码路径、docs、契约之间的映射 | `.project-context/metadata/*.json` |

注意：这是摘要沉淀，不是把 `docs/` 原文搬进 `.project-context/`。

### Stage 6：开发与联调

开发阶段主要维护：

```text
docs/<业务域>/<模块>/QA待确认.md
docs/<业务域>/<模块>/联调记录.md
.project-context/active-work/current.md
.project-context/active-work/blockers.md
```

QA 规则：

- `QA待确认.md` 只保存未决问题和确认过程。
- 问题确认后，必须回写 PRD、设计或契约。
- QA 文档不能成为长期唯一权威来源。

### Stage 7：验收与归档

阶段完成后可补充：

```text
docs/<业务域>/<模块>/验收记录.md
docs/<业务域>/<模块>/复盘.md
```

处理规则：

- 已完成事项从 `.project-context/active-work/current.md` 移除。
- 已解决阻塞从 `.project-context/active-work/blockers.md` 移除或标记关闭。
- 长期有效经验沉淀到 `.project-context/modules/`、`contracts/` 或 `decisions/`。
- 临时过程记录不进入长期上下文。

## 3. 不同文档的权威来源

| 信息类型 | 权威来源 | `.project-context/` 的作用 |
|---|---|---|
| 产品需求完整描述 | `docs/**/*PRD.md` | 摘要关键规则和边界 |
| 子模块差异规则 | `docs/<业务域>/<模块>/<子模块>.md` | 摘要不可破坏差异 |
| 技术实现方案 | `docs/**/*设计.md` | 摘要长期架构约束 |
| 接口字段和行为 | OpenAPI / Java Controller / 接口设计文档 | 记录跨模块语义和兼容性约束 |
| 数据表结构 | migration / 数据库设计 / Java Entity | 记录共享字段语义和状态约束 |
| 权限码 | 菜单配置 / 后端权限注解 / 权限设计 | 记录跨端权限约定 |
| 当前进度 | `.project-context/active-work/current.md` | 权威来源 |
| 阻塞风险 | `.project-context/active-work/blockers.md` | 权威来源 |
| 架构决策原因 | `.project-context/decisions/*.md` | 权威来源 |
| 代码事实 | 代码本身 / CodeGraph | 不重复记录 |

## 4. `/context-sync` 同步入口

项目提供 `/context-sync` skill，用于把 `docs/`、对话确认事项和代码变更同步沉淀到 `.project-context/`。

以下情况应执行 `/context-sync`：

- PRD 或设计已经确认，需要沉淀模块职责、边界、关键规则或已知坑点。
- 新增、删除或变更 API、数据库、事件、权限、状态枚举等跨模块契约。
- 修改了模块边界、跨子项目依赖或架构决策。
- 对话中确认了长期有效规则、风险、阻塞或协作约定。
- 一轮开发、联调或验收结束，需要同步文档、代码和上下文。
- Stop hook 提示存在可能需要同步的确认节点。

## 5. agent 开工顺序

处理需求或改代码时，推荐顺序：

1. 先读 `.project-context/INDEX.md` 判断目标模块和相关契约。
2. 读取对应 `.project-context/modules/` 和 `contracts/`。
3. 按需读取 `docs/` 中完整 PRD、设计或 QA。
4. 使用 CodeGraph 查询真实代码结构、入口和影响面。
5. 修改代码或生成文档。
6. 收工前同步必要的 `docs/`、`.project-context/` 和生成物。
