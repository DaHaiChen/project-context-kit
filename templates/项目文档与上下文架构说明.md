# 项目文档与上下文架构说明

## 说明

协作上下文中枢是一套面向多人、多子项目、多 agent 协作的轻量上下文架构，由 docs完整文档层、.project-context 语义索引层、CodeGraph 代码事实层和 ProjectContext MCP 同步校验层组成。 

## 1. 背景

本项目根目录用于统一维护多子项目、多 agent 协作时需要的文档、上下文、语义图谱和工具规则。

早期实践中，业务 PRD、设计方案、接口约定、QA、流程图等文档通常直接落在具体子项目的 `docs/` 目录下。这种方式适合单项目、单团队推进，但当项目逐步出现多个端、多个子项目、多个 agent 协同时，会出现以下问题：

- 文档分散在不同子项目中，agent 不容易判断应该先读哪份。
- PRD、设计、QA、临时讨论和最终规则混在一起，生命周期不清晰。
- 跨模块接口、数据库字段、权限、事件等契约容易被遗漏。
- agent 每次开工都需要重新理解项目背景，长期上下文容易丢失。
- 对话中确认的规则和决策如果没有沉淀，后续容易重复讨论或重复踩坑。

因此，本项目采用一套轻量的三层上下文架构：

```text
docs/               完整交付文档层
.project-context/   AI 维护的项目语义索引与契约层
CodeGraph           代码事实层
```

这套架构的目标不是增加文档负担，而是让“人维护完整文档，AI 维护压缩上下文”，并在必要节点同步。

## 2. 总体架构

```text
人 / 产品 / 研发
  |
  | 维护完整交付文档
  v
docs/
  PRD、需求共识、设计方案、接口设计、QA、验收、复盘
  |
  | PRD/设计确认、契约变化、模块边界变化、开发联调结束后执行 /context-sync
  v
.project-context/
  模块边界、契约摘要、长期约束、已知坑点、架构决策、当前阻塞、metadata
  |
  | Project Context MCP 生成
  v
generated/
  project-tree.md、project-graph.json、project-graph.mmd、stale-report.md

代码仓库 / 子项目
  |
  | CodeGraph 读取真实代码事实
  v
CodeGraph
  文件、符号、调用关系、影响面
```

## 3. 各层职责

### 3.1 `docs/`：完整交付文档层

`docs/` 是人主要维护的文档入口。

保存内容包括：

- 需求共识。
- 大模块 PRD。
- 子模块 PRD。
- 一期设计 / 技术设计。
- 接口设计 / Java 对接说明。
- QA 待确认。
- 联调记录。
- 验收记录。
- 复盘。
- 版本计划。

推荐结构：

```text
docs/
  README.md
  文档生成流程.md
  业务域/
    模块/
      需求共识.md
      模块PRD.md
      模块一期设计.md
      子模块A.md
      子模块B.md
      接口设计.md
      QA待确认.md
      验收记录.md
      复盘.md
```

`docs/` 的特点：

- 内容完整。
- 面向产品、研发、测试、评审人员和 agent。
- 可以包含过程、背景、细节、表格、流程图和完整设计。
- 是 PRD、设计和 QA 的主要权威来源。

### 3.2 `.project-context/`：项目语义索引与契约层

`.project-context/` 默认由 AI 维护。

它不替代 `docs/`，只保存长期有效、对后续协作有价值的压缩上下文。

保存内容包括：

- 模块职责。
- 模块不负责的范围。
- 关键流程和状态机摘要。
- 跨模块 API、数据库、事件、权限契约摘要。
- 长期有效的业务规则。
- 已知坑点。
- 架构决策原因。
- 当前重点、阻塞和风险。
- 文档、模块、代码路径和契约之间的 metadata 映射。

推荐结构：

```text
.project-context/
  INDEX.md
  architecture.md
  glossary.md

  modules/
    admin-web/
    java-backend/
    mini-program/
    app/
    testing/

  contracts/
    api.md
    database.md
    events.md
    permissions.md

  decisions/
  active-work/
  agent-notes/
  metadata/
  generated/
```

`.project-context/` 的特点：

- 内容短。
- 面向 AI、agent、新人和跨模块协作者。
- 重点记录“边界、契约、为什么、不能破坏什么”。
- 不搬运完整 PRD 和长篇设计。
- 从 `docs/`、对话确认、代码事实和 MCP 工具结果中抽取。

### 3.3 CodeGraph：代码事实层

CodeGraph 负责读取真实代码事实，包括：

- 文件。
- 符号。
- 函数、类、方法。
- 调用关系。
- 依赖关系。
- 影响面。

CodeGraph 不负责业务语义。它回答“代码现在是什么”；`.project-context/` 回答“业务和架构为什么这样”；`docs/` 回答“完整设计是什么”。

### 3.4 Project Context MCP：同步与校验工具层

Project Context MCP 用于辅助维护 `.project-context/`，主要能力包括：

- 根据文件列表或 diff 推断受影响模块和契约。
- 搜索 `.project-context` 中的语义文档。
- 读取指定模块上下文。
- 生成项目图谱。
- 检查 metadata、文档和 generated 是否过期或不一致。

常用工具：

- `project_context_sync_docs_from_diff`
- `project_context_map_files`
- `project_context_search`
- `project_context_get_module`
- `project_context_generate_graph`
- `project_context_check_staleness`

### 3.5 `/context-sync` skill：确认节点同步入口

`/context-sync` 是项目级 skill，用于在确认节点把 `docs/`、对话结论和代码变更同步沉淀到 `.project-context/`。

现在采用轻量模式，不是每次文档变化都执行。

应执行 `/context-sync` 的情况：

- PRD 或设计已经确认，需要沉淀模块职责、边界、关键规则或已知坑点。
- 新增、删除或变更 API、数据库、事件、权限、状态枚举等跨模块契约。
- 修改模块边界、跨子项目依赖或架构决策。
- 对话中确认长期有效规则、风险、阻塞或协作约定。
- 一轮开发、联调或验收结束，需要同步文档、代码和上下文。

不需要执行 `/context-sync` 的情况：

- 只是 PRD 草稿调整。
- 只是临时讨论。
- 只是还未确认的方案探索。
- 只是单份 `docs/` 文档内部措辞修改，且不影响长期规则和契约。

### 3.6 Stop hook：轻量提醒机制

Stop hook 会在会话结束时检测可能相关的变更，并提醒是否需要同步。

当前定位是提醒，不是强制。

如果检测到 `docs/`、代码、`.project-context/`、契约类文件或协作配置变更，会提示：

- 如果只是 `docs/` 草稿变化，可以只保留在 `docs/`。
- 如果 PRD / 设计已确认，或涉及 API、数据库、事件、权限、状态枚举、模块边界、架构决策，应执行 `/context-sync`。
- 如果执行了 `/context-sync`，应生成项目图谱并检查 stale。

## 4. 推荐工作流

### 4.1 日常需求阶段

```text
1. 在 docs/ 中写需求共识、PRD、设计、QA。
2. 草稿阶段不强制同步 .project-context/。
3. 用户、产品、研发继续评审和修改 docs/。
```

此阶段重点是完整表达需求，不增加上下文维护负担。

### 4.2 PRD / 设计确认后

```text
1. 执行 /context-sync。
2. AI 从 docs/ 抽取模块职责、边界、关键规则、已知坑点。
3. AI 更新 .project-context/modules/。
4. 如涉及 API、数据库、事件、权限，更新 .project-context/contracts/。
5. 如涉及重要取舍，更新 .project-context/decisions/。
6. 更新 metadata。
7. 生成 project graph。
8. 检查 stale。
```

此阶段 `.project-context/` 成为后续开发和多 agent 协作的压缩上下文。

### 4.3 开发前

agent 推荐读取顺序：

```text
1. .project-context/INDEX.md
2. 相关 .project-context/modules/**/*.md
3. 相关 .project-context/contracts/*.md
4. 相关 docs/**/*.md
5. 目标子项目 CLAUDE.md
6. CodeGraph 查询真实代码
```

### 4.4 开发后

如果只是普通实现细节变化，不一定更新 `.project-context/`。

如果发生以下变化，则执行 `/context-sync`：

- 改了模块边界。
- 改了 API 契约。
- 改了数据库字段、表、状态枚举。
- 改了事件、权限、跨模块依赖。
- 发现长期风险或坑点。
- 一轮联调或验收后形成长期结论。

## 5. 权威来源关系

| 信息类型 | 权威来源 | `.project-context/` 的角色 |
|---|---|---|
| 完整 PRD | `docs/**/*PRD.md` | 摘要关键规则和模块边界 |
| 需求共识 | `docs/**/需求共识.md` | 摘要目标、范围和不做项 |
| 技术设计 | `docs/**/*设计.md` | 摘要长期架构约束 |
| QA 未决问题 | `docs/**/QA待确认.md` | 已确认后沉淀为规则或阻塞 |
| API 行为 | 接口设计 / OpenAPI / Java Controller | 摘要跨模块接口语义 |
| 数据表结构 | migration / 数据库设计 / Entity | 摘要共享字段和状态语义 |
| 权限码 | 权限设计 / 菜单配置 / 后端注解 | 摘要跨端权限约定 |
| 架构决策原因 | `.project-context/decisions/*.md` | 权威来源 |
| 当前阻塞 | `.project-context/active-work/blockers.md` | 权威来源 |
| 代码事实 | 代码本身 / CodeGraph | 不重复记录 |

当 `docs/` 与 `.project-context/` 冲突时：

1. 以 `docs/` 中最新确认版本为准。
2. 由 AI 更新 `.project-context/`。
3. 如果冲突来自代码真实实现，还应结合 CodeGraph 和代码确认。

## 6. 优势

### 6.1 人只维护一份完整文档

人主要维护 `docs/`，不需要手动维护两套文档。

`.project-context/` 由 AI 在确认节点自动抽取和维护，减少重复劳动。

### 6.2 降低 agent 重新理解成本

agent 开工时不需要每次从大量 PRD 中重新总结项目背景。

`.project-context/` 提供压缩后的模块边界、契约、已知坑点和影响范围。

### 6.3 更适合多子项目协作

当项目同时存在 Web、Java 后端、小程序、App、后台任务等多个子项目时，`.project-context/` 可以统一记录跨项目关系和契约。

### 6.4 契约变化更容易被发现

API、数据库、事件、权限、状态枚举等跨模块契约集中沉淀，后续修改时更容易检查影响面。

### 6.5 对话中的长期结论不会丢

很多真实规则是在对话中确认的。通过 `/context-sync`，AI 可以把这些规则沉淀到 `decisions/`、`contracts/`、`modules/` 或 `active-work/`。

### 6.6 支持图谱和过期检查

Project Context MCP 可以生成项目图谱，并检查 metadata、模块文档、契约文档和 generated 是否不一致。

### 6.7 可以轻量使用

当前已经降级为轻量流程：草稿阶段只写 `docs/`，确认节点再同步 `.project-context/`，不会每次文档变化都触发重流程。

## 7. 劣势与成本

### 7.1 初期看起来偏重

在项目还没有真实代码仓库、模块数量少、只有少量 PRD 时，`docs/`、`.project-context/`、metadata、generated、skill、hook 这一整套会显得复杂。

### 7.2 需要同步纪律

如果 PRD 已确认但长期不执行 `/context-sync`，`.project-context/` 会落后于 `docs/`。

如果 `.project-context/` 落后，agent 可能读到过期边界或契约。

### 7.3 AI 抽取可能不准确

AI 可能：

- 把草稿当成确认事实。
- 抽取遗漏关键规则。
- 把不该长期保存的临时讨论写入上下文。
- 对契约影响面判断不完整。

因此关键契约和架构决策仍需要人工审核。

### 7.4 根目录不是 git 仓库时自动检测能力有限

当前根目录不是 git 仓库时，Stop hook 很难自动发现根目录 `docs/` 的所有变化。

这种情况下需要通过明确指令或文件列表执行同步。

如果未来希望完整使用自动检测，建议根目录纳入 git 管理。

### 7.5 metadata 需要维护

`.project-context/metadata/*.json` 是 MCP 和 agent 的机器索引。如果 metadata 没更新，图谱和影响面判断会变弱。

虽然它由 AI 维护，但仍需要 stale 检查辅助发现问题。

### 7.6 不适合所有项目

对于非常小、一次性、单模块、单人维护的项目，这套架构可能明显过度。简单 `docs/ + CLAUDE.md` 即可。

## 8. 适用场景

适合使用这套架构的场景：

- 多子项目协作。
- 多端协作，例如 Web、Java 后端、小程序、App。
- 文档会持续增长。
- PRD、设计、QA、接口约定经常变更。
- 多个 agent 或多人接力开发。
- 需要长期保存模块边界、契约和架构决策。
- 希望 AI 后续能自动理解项目上下文。

不适合或可降级使用的场景：

- 单人一次性项目。
- 没有跨模块契约。
- 只有少量 PRD。
- 不需要 AI 长期维护上下文。
- 没有持续开发和联调过程。

## 9. 当前采用的折中方案

本项目当前采用“架构保留，执行降级”的方案：

```text
草稿阶段：只写 docs/
确认节点：执行 /context-sync，同步 .project-context/
开发前：先读 .project-context，再按需读 docs/ 和代码
开发后：只有影响契约、边界、决策或长期坑点时才同步
```

这样保留长期协作能力，同时避免每次文档草稿变化都触发复杂流程。

## 10. 最小使用指南

### 写新模块 PRD

```text
1. 在 docs/<业务域>/<模块>/ 下写需求共识、PRD、设计、QA。
2. 草稿阶段只维护 docs/。
3. PRD 或设计确认后，说“这个模块已确认，执行 /context-sync”。
4. AI 抽取长期上下文到 .project-context/。
```

### 修改契约

```text
1. 在 docs/ 或代码中修改接口、数据库、事件、权限等内容。
2. 执行 /context-sync。
3. AI 更新 contracts、modules、metadata。
4. 生成图谱并检查 stale。
```

### 普通实现改动

```text
1. 读取 .project-context 判断模块边界。
2. 按需读取 docs 获取完整设计。
3. 使用 CodeGraph 查代码。
4. 修改代码。
5. 如果没有影响契约或长期规则，不必同步 .project-context/。
```

## 11. 结论

这套架构不是为了替代 `docs/`，也不是要求每次都走重流程。

它的核心价值是：

```text
人维护完整交付文档。
AI 在确认节点维护压缩上下文。
CodeGraph 提供真实代码事实。
Project Context MCP 负责生成图谱和检查一致性。
```

如果按重流程使用，它会显得多余；如果按轻量方式使用，它可以在不增加太多日常负担的前提下，为后续多项目、多 agent、长期协作提供稳定上下文。
