# Project Context 目录维护说明

本文用于向团队解释 `.project-context/` 为什么要这样拆分，以及每类人员应该维护哪些目录。

## 总览

```text
modules/     写“模块为什么这样工作”
contracts/   写“模块之间怎么承诺”
decisions/   写“为什么选这个方案”
metadata/    写“工具怎么找到它们”
generated/   放“工具生成的全局图谱”
active-work/ 写“现在正在发生什么”
agent-notes/ 写“临时交接信息”
INDEX.md     写“从哪里开始读”
```

这套结构的目的不是增加文档负担，而是把不同生命周期、不同用途的信息分开放：长期设计、跨模块契约、短期协作、机器索引和自动生成物不要混在一个文件里。

## 1. `modules/`

**谁维护：**

- 负责具体业务模块的人。
- 例如搜索模块负责人维护 `modules/search.md`，权限模块负责人维护 `modules/permission.md`。

**维护什么：**

- 模块职责。
- 模块不负责什么。
- 关键业务流程。
- 业务规则。
- 依赖哪些模块。
- 修改这个模块时要注意什么。

**为什么放这里：**

CodeGraph 能告诉 AI “函数怎么调用”，但不能稳定告诉 AI “这个模块为什么这样设计、边界在哪里”。`modules/` 是 AI 和新人理解业务模块的第一入口。

## 2. `contracts/`

**谁维护：**

- 负责跨模块接口的人。
- API 由后端或接口负责人维护。
- 数据库契约由数据模型或后端负责人维护。
- 事件契约由异步链路的生产者和消费者共同维护。

**维护什么：**

- API 语义。
- 数据表共享字段语义。
- 事件消息格式。
- 哪些字段不能随便改。
- 哪些变更会影响其他模块。
- 契约的权威来源，例如 OpenAPI、schema、proto。

**为什么放这里：**

契约是跨模块协作的边界。改代码最容易出问题的地方是“我只改了自己模块，但破坏了别人依赖的语义”。`contracts/` 让 AI 在改接口、数据库、事件前知道影响面。

## 3. `decisions/`

**谁维护：**

- 做出架构决策的人。
- 技术负责人、模块负责人，或者当次做出长期设计选择的 agent。

**维护什么：**

- 为什么选这个方案。
- 放弃了哪些方案。
- 这个决策带来的约束。
- 以后改动时不能忽略的背景。

**为什么放这里：**

代码只能说明“现在是什么”，很难说明“为什么不是别的方案”。架构决策如果不记录，后续 AI 或新人容易反复推翻已有判断。`decisions/` 用来防止重复踩坑和反复争论。

## 4. `metadata/`

**谁维护：**

- 通常由工具或 MCP 自动维护。
- 人可以少量手改，例如新增模块时添加模块到代码路径的映射。

**维护什么：**

- 模块对应哪些文档。
- 模块对应哪些代码路径。
- 模块依赖哪些契约。
- 模块负责人是谁。
- 外部系统来源有哪些。

**为什么放这里：**

`.md` 适合人读，但不适合工具稳定解析。MCP 需要一个机器可读入口，快速判断：这个文件属于哪个模块、这个模块要读哪些文档、改这个代码要检查哪些契约。`metadata/` 是工具索引层，不是主要给人写长文的地方。

## 5. `generated/`

**谁维护：**

- 工具自动生成。
- 人一般不要手写。

**维护什么：**

- `project-graph.json`
- `project-tree.md`
- `project-graph.mmd`
- `stale-report.md`

**为什么放这里：**

这些文件是从 `modules/`、`contracts/`、`decisions/`、`metadata/` 合成出来的结果，用来快速看全局图谱和检查文档是否过期。因为它们是生成物，所以单独放，避免和人工维护文档混在一起。

## 6. `active-work/`

**谁维护：**

- 当前正在做跨模块工作的 agent 或负责人。

**维护什么：**

- 当前正在推进的事情。
- 当前阻塞。
- 短期风险。
- 下一个 agent 开工前必须知道的信息。

**为什么放这里：**

这是短期协作状态，不适合写进长期架构文档。例如“这周搜索排序正在重构，先不要改 ranking API”，这种信息应该放在 `active-work/`。

## 7. `agent-notes/`

**谁维护：**

- 每个完成任务、需要交接上下文的 agent。

**维护什么：**

- 交接说明。
- 临时发现。
- 尚未沉淀成长期文档，但下一个 agent 需要知道的信息。

**为什么放这里：**

避免所有临时信息都污染正式文档。如果某条 agent note 反复有价值，再沉淀到 `modules/`、`contracts/` 或 `decisions/`。

## 8. `INDEX.md`

**谁维护：**

- 项目上下文维护者。
- 或者新增模块、契约的人顺手更新。

**维护什么：**

- 项目目标。
- 阅读顺序。
- 模块地图。
- 契约地图。
- 关键入口。

**为什么放这里：**

这是 AI 和人进入 `.project-context/` 的首页。它不应该写太多细节，只负责导航。

## 9. `architecture.md`

**谁维护：**

- 架构负责人。
- 或者做跨模块架构变更的人。

**维护什么：**

- 系统整体架构。
- 核心链路。
- 模块边界总览。
- 全局技术约定。

**为什么放这里：**

它回答的是“整个项目怎么组成”，不是某个模块的细节。AI 在做跨模块改动前应该先读它。

## 10. `glossary.md`

**谁维护：**

- 业务负责人、产品、后端，或任何发现术语歧义的人。

**维护什么：**

- 业务术语。
- 技术术语。
- 缩写。
- 同名但不同义的概念。

**为什么放这里：**

术语不统一会导致 AI 和人都误解需求。例如“节点”“实体”“知识点”“文档块”这类词，最好统一定义。

## 维护原则

- 长期有效的信息放 `modules/`、`contracts/`、`decisions/`、`architecture.md`。
- 短期协作状态放 `active-work/`。
- 临时交接信息放 `agent-notes/`。
- 机器可读索引放 `metadata/`。
- 自动生成结果放 `generated/`。
- 不要把密钥、token、cookie、个人隐私或生产敏感数据写入 `.project-context/`。
